What's New?

The following new features were added between 0.2.61 and 1.0.65:

Sandbox Mode Only

  • Sample Code and Developer Applications - Several open source sample applications were released and added to the developer documentation. Included is sample code from the Data Browser, Dashboard, Ingestion Validation Tool, and the Ingestion Client Batch Tool for Windows. These tools can be downloaded from the SLC Github repository at https://github.com/slcedu. For more information on these tools, please see the SLI Application Development: Sample Applications and Code section of the developer documentation.
  • Auto-Loading a Sample Dataset when Provisioning a Landing Zone - Allows sandbox user to select between small/medium datasets to automatically load once their landing zone is provisioned.
  • Realm Selection and User Emulation - Sandbox authentication and login has been simplified to improve the application developer user experience. When logging in to a sandbox application, developers are now presented with a simple page allowing them to administer their account or test their applications. Both options require that developers log in with their sandbox credentials.

    If the developer chooses to test an application, the developer will be taken to the impersonation screen. Here they can choose the manual user specification route that existed previously, or the developer can select a user from one of the sample datasets that were provided when the developer provisioned the account. Note that if the developer didn't choose to use the provided data set, they must use the manual option.

All Modes

  • URI rewrites for Context Resolution Pre-Filtering - An implied context is added to open-ended URIs that pre-filters the results. Context is analogous to an ID as the second parameter of a URI. The context used for all of these URIs is documented in the API reference information in the developer guide and can be seen in the HTTP header in the response. Application developers need to update their applications to work within the context of the current user's current and direct associations. To return results that deviate from the implied context, you must explicitly provide additional context in the URI.

    Example: As staff with mutlple schools, /schools does not return all of the schools you can see. Instead, /schools only returns the schools that you have a current and direct association with. You can use /schools/{IDs}/sections for sections from a specific school instead of /schools/sections for sections from all of the schools you have a current and direct association with.

  • Added JavaScript SDK - A JavaScript SDK has been added for the SLI. The SLI JavaScript SDK uses the server-side JavaScript environment Node.js.
  • Ingestion Performance Improvements: Referencial integrity no longer enforced - During Ingestion, the system no longer enforces that references in an entity refer to an existing entity. If the data being ingested has incorrect or incomplete references, these are not reported. The data is stored as presented. For example, if the SEA Reference within an LEA fails to resolve, nothing is reported so be aware an incomplete or broken EdOrg hierarchy is possible. Referential integrity has been removed with a few exceptions:
    • Referencial integrity is still applied for the gradingPeriod entity.
    • All transformed entities use queries during the transformation process to ensure referential integrity. These entities include:
      • Assessment
      • StudentAssessment
      • AttendanceEvent

DE1759: Open Source Dashboard fails on Tomcat

The open source dashboard did not work when run on Tomcat. The workaround was to use Jetty to run the open source dashboard.

DE215: Sorting in App Registration Tool

The app developer could not sort the applications in the application registration tool.

DE1679: SEA/LEA Federated End User Can Have Only One Role

An SEA/LEA end user could have multiple user roles mapped to multiple SLI roles. The system gave the user permissions that are equal to a union of his/her list of roles. The system now limits the user to the proper context for his/her role.

DE1825: POSTing to a studentSectionAssociation without a beginDate causes a 500 error

POSTing to a studentSectionAssociation without a beginDate causes a 500 error. 400 validation error would be expected.

DE1606: Admin application returns generic errors when realm is not created.

While onboarding in Production, if a realm had not yet been created for the district, but the user tried to access the admin tool, the user did not receive proper error messaging.

DE1571: Sandbox: can't disable application's data access to EdOrgs

In Sandbox, if a user enabled an app for a certain EdOrg, he/she cannot revoke the access through the admin interface in the portal.

DE1920: Editing of schools not limited by context

Because the EdOrgs are global, everyone with a role that has the Write Public right could modify any EdOrg. This has since been restricted.

DE1284: Data Browser should deny use to users who don't have admin right

Users that don't have the IT Administrator role (both in Sandbox & Production) cannot access the Admin page of the Portal. They cannot access the Data Browser. However, if they knew the direct URL to the data browser, they were able to access the app. The user will now be denied access.

DE1550: Ed org stamper not properly stamping cohorts

Teacher/Staff cannot access Program/Cohort that is associated with an EdOrg above them, even though they are directly associated with it.

DE1688: Using a negative limit causes a 500 error in API

If the limit was set to a negative value on an API call (e.g. ?limi t= -1), a 500 error was returned. Now a 400 error is returned.

DE910: Application registration EdOrg display needs to handle more than 50 EdOrgs

The list of LEAs in the enablement part of application registration tool (both in Production and Sandbox) did not list more than 50 LEAs. The system now supports a list of more than 50 EdOrgs.

DE1932: Negative offset on paging causes an error

Negative offsets in relation to paging are now handled properly.

DE1948: Can't walk down the EdOrg hierarchy through hateos links

HATEOAS links for child EdOrgs were missing, which made the user unable to traverse down the hierarchy to lower-level EdOrgs.

DE2047: Stale databrowser cookie causes failure of databrowser

In the Sandbox in impersonation mode, if the developer wanted to test the databrowser with 2 different users, the Data Browser would fail. He/she will have to delete the cookie to be able to do the switch.

DE2055: Long-Lived session token generator not updated to work with db-per-tenant

The long-lived session token generator did not work with db-per-tenant. The script needs to be updated to work with the new data layout.

DE2114: Linda kim can not access staffEducationOrgAssignmentAssociations

The teacher got an error to accessstaffEducationOrgAssignmentAssociations.

DE2088: Learning Objectives resource broken

The /learningObjectivesresource returned an error. It now responds properly.

DE1906: Limit behavior changed for some entities between 4.7 and 6.1

For some entities, the number of results returned did not honor the offset and limit request. This occurred in both sandbox and production.

DE2035: Sandbox impersonation log in is forced every time

In Sandbox, the user was asked to re-log in every time he/she switches the user that he/she was impersonating.

DE1970: 500 errors after creating and reading cohorts for teacher

Unexpected 500 and 403 errors were returned when a teacher tried to access cohorts.

DE2090: Assessments resource broken

The /assessments resource returned an error. It now responds properly.

DE2091: CompetencyLevelDescriptor & CompetencyLevelDescriptorTypes resource broken

The resources have been removed.

DE2066: Adding excludeFields results in 500

Certain uses of excludeFields caused errors. These have been resolved. excludeFields can now also be used with includeFields. In this case, excludeFields takes precedence.

DE1757: Using includeFields query parameter on PII fields results in response not being unencrypted

When a user passed in an includeFields query parameter on a PII field, the results returned were still encrypted. Now, whenever includeFields is used, the type and metadata are also included.

DE1911: /courseTranscripts API endpoint returns records of type studentTranscriptAssociations

A naming convention mismatch relating to the /courseTranscripts API endpoint returned records of type studentTranscriptAssociations. It now returns courseTranscripts with type courseTranscript.

DE2030: First time using portal SEA/LEA admins have no admin apps

In production mode SEA admin users did not see any administrative applications in the portal in the onboarding process until they had provisioned a landing zone.

DE1764: Implement missing teacher context resolver for studentCompetencyObjectives

A security context resolver for Teacher (user) and studentCompetencyObjective (entity) was added to assure correct enforcement of security model business rules.

DE1969: Custom Roles can't identify Realm correctly and fails

Existence of multiple realms for a single EdOrg displayed unauthorized data, realms from other tenants and did not permit the creation of custom roles.

DE1910: Paging through some entities results in duplicate records

The API often returned duplicate records when using the paging functionality to page through a large result set. This occurs in both sandbox and production.

DE1873: Paging through students results in spurious 403 and 500 errors and incomplete results

The API often returned 403 or 500 HTTP error codes or incomplete results when using the paging functionality to page through a large result set. This occurred in both sandbox and production modes.

DE1823: Change uniqueSectionCode length in EdFi extension

The original field length was not large enough for system generated data. We applied an extension that keeps the EdFi limit to the original, but allows an ingestion for a size up to 48. If the actual length is greater than or equal to the EdFi definition, no error or warning is shown. If the length is greater than 30 but less than or equal to 48, then a warning is generated but ingestion continues. If the actual length is greater than 48 an error occurs.

DE1876: Posting an entity twice results in API 500

Posting an entity twice would result in an API 500 error. We've changed features of the natural keys calculations which caused this error. We have fixed this defect.

DE1722: API call - /api/rest/tenant returns all tenant information

In Sandbox and Production if a logged in developer using his/her token makes the /api/rest/tenant call, information for all of the tenants will be returned.

DE921: Duplicates are not reported

Error reports are not always clear regarding the records persisted. When there are duplicates within the XML file, or a record already exists, the ingestion engine will still update the record. However, this record count is not included in the Ingested records count.

DE1458: Handle more than one teacher of record

The ingestion service can currently accept more than one teacher for a single section. The dashboard does not currently support this behavior.

DE157: School and EdOrg only supports a single representation of parentEducationAgencyReference

Reference resolution is only supported as the default scenario that features no exceptional or error conditions. As a result, a user may ingest valid EdFi format with improper references that would fail to resolve.

DE325: Global uniqueness of ProgramIdentifier is assumed when multiple EdOrgs can use the same IDs

Ingestion process assumes that the Program ID is unique within a tenant, but this should be unique within the specified EdOrg instead. As a result, the reference could resolve incorrectly.

DE273: DisciplineDescriptor not currently supported in Interchange-StudentDiscipline

This descriptor is not currently supported. However, a reference to this descriptor is required for the DisciplineAction entity.

DE198: EducationOrganization references to Program not properly ingested

This reference hasn't been implemented yet. As a result, the API won't provide a link from the EdOrg to the Program.

DE1434: Dashboard Builder - Drag and drop on release doesn't always return the page to the right state

When a large dataset is available, the initial dashboard view may only display a subset of the data. The dashboard currently does not display a message explaining that only a subset of data is displayed.

DE1647: Dashboard may not display all data

When a large dataset is available, the initial dashboard view may only display a subset of the data. The dashboard currently does not display a message explaining that only a subset of data is displayed.

DE1457: Section Profile - Handle 403 error when making teacher call

Currently, making a teacher call for a section with two "teachers of record" returns a 403 error. This error can cause the sectionInfo panel to not display.

DE1832: Dev without "Ingestion User" role can still ingest data

An application developer has access to the Landing Zone provisioning tool, and can provision a landing zone but can not access it via SFTP. Therefore can not ingest data manually but can do so through auto provisioning.

DE1573: Remove Attendance Rate

The current attendence record includes an attendance rate column that may not be accurate for some use cases. The column will be removed by default in a future release.

DE1760: Including the word Script in data results in blacklist filter error.

If your data includes the word "script," then the user will be added to a blacklist filter.

DE861: Super Admins should not be capable of ingesting EdOrgs above their own EdOrg

During ingestion process the landing zone is tolerant of any EdOrg within a tenant being ingested.For example, two LEAs belong to the same SEA. LEA #1 can ingest data for LEA #2 since they are both part of SEA. LEA #1 can also ingest data for SEA. This is not ideal since data specific to each LEA are not restricted, but it is functional and ingestion works properly.

DE162: Course name (not course code) is used to uniquely identify and refer to Course entities

Course entities are uniquely identified by course name within an education organization, rather than by course code.

DE710: Section, school and session references in AttendanceEvent are ignored

The way AttendanceEvent entities are handled ignores explicit characterization of the attendance event - by school, section, session. As a result, AttendanceEvents for students attending multiple schools will not properly persist.

DE2156: Response data for API calls to api/rest/adminDelegation/myEdOrg contain full entity data, not just the body

The API call that gives an info about what has been delegated by an LEA (api/rest/adminDelegation/myEdOrg) reveals unnecessary additional information.

DE2155: SEAs can't see delegation info when only security event viewing is delegated

The API call that returns what have been delegated (from the LEA) to SEA, gives no info (empty list) in a case when only the viewing of security events has been delegated to the SEA.

DE2007: Sandbox developer onboarding email does not reflect recent login changes

In the email that the developer gets when on-boarding on Sandbox, one of the steps instructs the developer to select a realm from the drop down realm chooser, instead of referencing the new realm page.

DE1908: Bad error message given illegal characters in Custom Roles

Both in Sandbox and Production, when illegal characters are added for a Role group, instead of returning appropriate message, an error is returned revealing code.

DE2160: Dashboard: Select filter after listing section removes all student data

When using SLC Dashboard and viewing a list of students, in some cases choosing a filter related to that student's "special needs" status will result in a "no students found" error.

DE1728: Portal error if OAuth Callback URL doesn't begin with https

If an application includes a callback URL that does not begin with, "https://" then an error message will be returned.

DE1871: Data migration scripts not completely migrating application data to new sandbox

A developer logs into the new sandbox in the sandbox environment realm and does not see their application. Workaround: The developer can see the application in the SLC realm. The developer can then click the edit button and re-save the application so it appears in the sandbox realm.

DE2173: Truncation limit for ingestion errors and warnings misconfigured in SLI Properties

Each error file is truncated after the maximum number of warnings and the maximum number of errors have been reached. The maximum number of errors is 10000 per input file. The maximum number of warnings is 10000 per input file.

DE2004: Custom Roles Migration

Custom roles has changed to support a new "isAdminRole" right. As a result, you must migrate to the new custom roles framework by running a script on the command line. The script can be called using the following command: SLI/tools/customRolesMigration.rb

DE2294: Applications become unavailable to users after purge and reload of data set

After purge and reload of data, you must reenabled access for the desired EdOrgs and then application registrations must be saved again.
