Alpha.0 is the initial release of the SLC technology services, education dashboards, and documentation.

With this release, SLC provides application developers with access to a sandbox SLC environment and a series of dedicated developer camps taking place over the next several months [[[FIXME: link "developer camps" to the corresponding URL for info]]]. While accessing their sandbox, developers can ingest SLC sample data sets, sign in as different users with varying roles, and create new applications that interact with the data using the SLC REST API. These applications can perform the same tasks available to state and district tenants.

REMINDER: The Sandbox MUST NOT contain actual student data. Users should load only artificial or fabricated data into the sandbox. The SLC provides sample fabricated data sets available for download from the Sandbox Developer Web Site at http://dev.slcedu.org/getting-started/sandbox/.

DE563: INGESTION: Some uses of IDREF are not supported

InterchangeEducationorganization/School/ClassPeriodReference and InterchangeEducationorganization/School/LocationReference among others are currently not supported by IDREF resolution.

DE157: INGESTION: School and EdOrg only supports a single representation of parentEducationAgencyReference

Reference resolution is only supported as the default scenario that features no exceptional or error conditions. As such a user may ingest valid EdFi format with improper references that would fail to resolve.

DE325: INGESTION: Global uniqueness of ProgramIdentifier is assumed when multiple EdOrgs can use the same IDs

Ingestion process assumes that the Program ID is unique within a tenant, this should really be unique within the specified EdOrg. The potential is that the reference could resolve incorrectly.

DE273: INGESTION: DisciplineDescriptor not currently supported in Interchange-StudentDiscipline

Entity not being ingested but a reference to it may exist in another entity.

DE198: INGESTION: EducationOrganization references to Program not properly ingested

This is a reference which hasn't been implemented, the effect is that the API won't provide a link from the EdOrg to the Program.

DE641: SDK: Java client API library requires a SLIClient instance per user

The client library does not support multiple sessions. If used in a multi session environment as is, then users can have conflicting actions eg. update the same resource at the same time.Workaround:Create an instance of SLIClient for each connection to the API, then attach the SLIClient to the HttpSession. Sample code for this is available in the AuthFiter.java class for the Sample application.

DE356: DASHBOARD: Enrollment History - Teacher cannot see the names of past schools their students attended

In Dashboard's Enrollment History panel, teachers cannot see the names of past schools a student attended, even when they are in the same district. The dates of enrollment and other data do appear, but school name is blank.

DE230: DASHBOARD: Student column in list of students suggests that it's sorted when first loaded

Lists of students suggest that they are sorted alphabetically on load, however they are not. Pressing the normal sort button in the header works around this issue.

DE162: INGESTION: Course name (not course code) is used to uniquely identify and refer to Course entities

Course entities are uniquely identified by course name within an education organization, rather than by course code.

DE710: INGESTION: Section, school and session references in AttendanceEvent are ignored

The way AttendanceEvent entities are handled ignores explicit characterization of the attendance event - by school, section, session, so AttendanceEvents for students attending multiple schools will not properly persisted.

DE80: INGESTION: Some fields of LocalEducationAgency are not persisted

LEACategory,CharterStatus,LocalEducationAgencyReference,EducationServiceCenterReference fields of LocalEducationAgency entities are currently ignored.

DE820: API: Field validation enumerations are not quoted correctly in API responses

Some fields contain enumerated values with commas even though they are single values. For example, the course.academicSubject field contains this single enumeration: "Agriculture, Food, and Natural Resources." This appears to be three distinct enumerated values in a validation error message.

DE857: DATA BROWSER: No error is displayed when user traverses data they are not authorized to access

The Data Browser does not inform the user why there is no result (nothing is returned) when a user tries to navigate (click on URL) to data that he/she has no permissions to access.

DE858: DATA BROWSER: No notification message is displayed when accessing an empty list

If a user click a URL that returns an empty list, there is no message that states that. The user might get an impression that the URL is incorrect.

DE789: DATA BROWSER: Incorrect user name is displayed after re-login with different user (after session have expired)

If the user tries to re-log in in the Data Browser with a different user, after the session of the prior user was expired, the top right corner of the Data Browser will still show the old user's name.Workaround: Users can close and relaunch the browser to see correct name displayed.

DE754: ADMIN TOOLS: Admin app not handling API session timeout

In some cases, when the user tries to remove an application in the Application Registration Tool, the application is removed from the system, but it still appears in the application list. The user should refresh the page, so the application is removed from the list.

DE911: INGESTION: Irregular error logging when same data set is ingested multiple times

Occasionally Ingestion gives error report when there is no error.

DE921: INGESTION: Duplicates should be reported

Error reports are not always clear regarding the records persisted. When there are duplicates within the XML file, or a record already exists, ingestion engine will update the record, but this record count is not included in the Ingested records count.

DE212: INGESTION: Processed files are left in a hidden directory

Ingestion process leaves files in ingestion landing zone folder lz/inbound/.doneWorkaround: Manually delete files in .done folder

DE935: INGESTION: large ingestion in one tenant may delay visibility of new data in another tenant.

New data that is ingested or created through the API needs to be tagged before it will be visible to users. The time taken to tag new data is dependent upon how large the dataset is, how much of the data is new, and the state of the tagger when the new data is created. The tagging process runs across all tenants, implying that a large ingestion in one tenant may delay visibility of new data in not only that tenant, but in others as well.

DE861: INGESTION: Super Admins should not be capable of ingesting EdOrgs above their own EdOrg

During ingestion process the landing zone is tolerant of any EdOrg within a tenant being ingested. For example, two LEAs belong to the same SEA. LEA #1 can ingest data for LEA #2 since they are both part of SEA. LEA #1 can also ingest data for SEA. This is not ideal since there is not restriction to data specific to each LEA, but it is functional and ingestion works properly.

DE608: INGESTION: References to Course by other entities do not use CourseCode

Entities referencing Course use CourseTitle (not CourseCode and EdOrg as specified by Ed-Fi) to uniquely identify a Course entity. Users will see inconsistencies with local descriptions of Courses until the entity is properly handled.

DE85: INGESTION: CourseOffering is ingested in a way that deviates from Ed-Fi

The CourseOffering entity is ingested in a way that deviates from the Ed-Fi definition. Users may see inconsistencies with local descriptions of courses.

DE87: INGESTION: Section entity is ingested in a way that deviates from Ed-Fi

The Section entity deviates from the Ed-Fi definition and is related to a defect where CourseOffering is not handled properly. Users may see inconsistencies with local descriptions of courses and sections until the CourseOffering defect is resolved.

DE682: API: Duplicate links in certain types of resource responses.

In the response bodies of certain resources, links with identical names and results may appear. However, both forms of the duplicated URIs resolve correctly.

DE1072: INGESTION: Ingestion job will begin processing a new job even though previous job has not yet completed

Ingestion job will begin processing a new job even though previous job has not yet completed. Workaround: verify that a job is completed prior to uploading another job

DE828: INGESTION: Session entity Grading Period reference not fully supported

EducationOrganizationIdentificationCode is not supported as a form of reference.Workaround: UseStateOrganizationId as the reference insideGradingPeriodIdentity.

DE194: INGESTION: Not all .tmp files deleted from the user's landing zone after processing is complete

Workaround: users can delete the files manually.

DE694: SDK: Sample application should handle lack of access more elegantly than an ISE 500

This error occurs when the authenticated user doesn't have any contextual data or does not have rights to access the application.Workaround:Identify a user in the dataset ingested that has contextual data associated. Eg. a teacher with sections and students.SLC Operator has to enable the application for the user's Educational Organization.

DE1083: PORTAL: When a user reports a problem via portal no email is sent to SLC

Currently, using report a problem form in the portal does not send a problem report to SLC. Workaround: manually report a problem by sending email to issues@dev.slcedu.org

Known Limitations

Known limitations in the Alpha.0 release can be summarized as follows:

  • When sandbox users ingest data sets through the landing zone, they may experience longer-than-expected load times. To address this, we have set a size limit of 1 MB for ingestion archive files, corresponding to approximately 30-50MB of uncompressed XML data, or about 30,000-50,000 records. In Alpha.0, ingesting a data set of this size should take about 15 minutes. This load time issue first appeared during the initial transition to the Alpha.0 operational environment, and is currently under diagnosis. Please check the Sandbox Developer Web Site at http://dev.slcedu.org/getting-started/sandbox/ for daily updates on this issue.

  • Some Ed-Fi entities are not yet supported and, thus, not consumable through data ingestion. For additional information, see the SLC Data Ingestion White Paper from dev.slcedu.org/documentation.

  • Bulk data extracts are not yet supported.

  • Not all ingestible data is displayed in SLC educator dashboards, including discipline incidents, full attendance records, current courses, and grades.

  • Leader roles do not yet provide aggregate view of data.

US2844: Enable Sandbox users to switch between already provisioned LZ

In sandbox the user can only provision a landing zone for a particular educational organization once, which limits them from switching between landing zones for retesting with data that they used before. There is no limitation to the number of landing zones they can provision using a custom educational organization.Workaround:If the user wants to re-ingest a sample dataset and can not re-provision the landing zone, the best way is to create a custom landing zone and update the sample data to have the matching high level ed-org.

US2660: App Permissions - Assigning Existing Data Category Rights

An application that is authorized by a LEA, can access the LEA's data based on the user's permissions (on which behalf the application is accessing LEA's data) and the application features/capabilities. There is no additional restriction that LEA can impose other than 'blanket' access (application can or cannot access LEA's data on user's behalf). Additional functionality will be built to allow the LEA to choose what rights over the existing data categoriesto give to certain application. The intersection of the user's and application's permissions and the application's capabilitieswill define in which way the user will be able to access the LEA's data. For example, if an application allows editing of general student data, and a user with IT Administrator role (which gives permission for editing student data) is using the application, he/she will not be able to edit any student data if this right has not been given to the application by the LEA.

US1727: Stable URLs for Student Profile Pages

In Dashboard, student profile and list URLs are unique and can be bookmarked, but will change at a later time, and with the ingestion of new data, so they are not permanently canonical at this time.

US2724: Remove "Attendance and Discipline" Tab Until Data is Available

In some cases depending on configuration, a blank "Attendance and Discipline" tab may appear in Student Profiles.

US2786: Admin: Provision Landing Zone shouldn't require EdOrg Entry

When provisioning a landing zone, the user must provision the landing zone at the edOrg they are a part of.

US2714: Add Pagination to List of Students

Dashboard displays a maximum of 50 students in a selected section, and does not support pagination at this time.

US1030: Current Courses and Grades Panel

Student Profiles do not yet have a tab displaying a student's current courses and associated grades, even when this data is ingested into SLI.

US2121: Password Reset for SLI Hosted Users

If an SLI hosted directory user has forgotten own password, he/she will have to ask the SLC Operator (in an offline process) to reset the password for him/her. A feature will be added to the SLI Identity Providerthat will allow the SLI hosted directory users to reset their password.

US1032: Attendance History Panel in Student Profile

Attendance and discipline data are not yet available in Student Profiles.
